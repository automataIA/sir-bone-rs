# Contributing

## Development Setup

```bash
cargo fmt
cargo test
cargo clippy -- -D warnings
```

For supply-chain checks:

```bash
cargo deny check
```

## Change Expectations

- Keep changes scoped to the behavior being fixed or added.
- Add tests for permission, file mutation, network, session, and audit changes.
- Do not weaken existing tests to make a change pass.
- Avoid live network dependencies in tests unless they are explicitly ignored or documented as integration tests.
- Preserve the default offline behavior of `sirbone doctor`.

## Security-Sensitive Code

When touching tools, permissions, hooks, MCP, snapshots, web fetch/search, or session audit:

- prefer pure unit tests for policy decisions;
- route every mutating native file tool through `mutation_target`;
- keep protected `~/.sirbone/{system,prompts,skills}` and `~/.sirbone/config.json` writes gated;
- make denial reasons visible to the user and model;
- redact provider keys and tokens in errors.

## Release Checks

Before tagging:

```bash
cargo fmt --check
cargo test
cargo clippy -- -D warnings
cargo deny check
```

The release workflow is generated by `cargo-dist`; update `dist-workspace.toml` rather than hand-editing generated release logic unless the generated workflow must be regenerated.
